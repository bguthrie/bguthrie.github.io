# Introducing Prospero: A writing tool that argues back

## Thesis
Prospero is a Claude Code plugin that stages blog-post writing as a four-phase pipeline — interrogate, critique, author, revise — in which the AI interrogates your argument, steelmans the opposition, writes first-draft prose in your voice, and invites your edits. An independent critic agent runs in a fresh context window and produces a structured rubric-based review of the outline or draft, so the argument has to survive challenge before prose gets written, and again before publication. The tool is built for writers who find the author seat more stressful than the editor seat — the engineer-essayists who hem and haw for weeks and sometimes never publish at all. For fluent writers it does not accelerate the work; for paralyzed perfectionists it may unblock it. Either way, you stand behind what you publish with more confidence.

## Antithesis
The piece is a release announcement, and the audience is HN-technical and skeptical. Three objections will land first.

First, and most directly: **the prose machine is what AI writing tools are for.** The reader's mental model of an "AI writing tool" is ChatGPT generating 800 competent flavorless words from a prompt, or Cursor-for-prose autocompleting sentences. A tool that explicitly de-emphasizes prose generation looks like a category error. "If I'm still doing most of the writing, what am I paying for?"

Second: **this is just prompt engineering with extra steps.** A disciplined user of any chat interface could run the same phases manually — brainstorm the thesis, ask for steelmen, open a fresh window for oppositional critique, then ask for a draft. Why a tool? Why a plugin? What does tooling add that a thoughtful practitioner couldn't replicate?

Third: **the author is arguing against speed.** Every AI tool pitch in 2026 is about throughput. A tool that explicitly admits "it did not make me faster" is fighting the industry narrative and will be read as rationalizing a tool that did not deliver on the productivity promise.

Three secondary flags the piece should close before an HN commenter opens them: whether oppositional-critique writing tools are already a saturated category (Perspectives, DebateDevil, and other monetized apps exist in the neighborhood); whether the "LLMs are sycophantic in a single session" claim is offered as demonstrated fact rather than sourced bet; whether "Superpowers proved this pattern works" overstates what a community-adoption story can prove.

## Synthesis
The prose-machine objection misreads the pitch. Prospero does generate prose — the `/author` phase exists — but it generates prose *last*, after the argument has survived challenge, and treats that prose as a scaffold for editing. The position is not "AI should not write prose"; it is that prose is the cheap part, and tools that front-load it are optimizing the wrong layer. This extends the argument of "AI like a bicycle" — amplify intent, do not substitute it — into writing.

The prompt-engineering-with-extra-steps objection mistakes ritual for structure. Two properties of Prospero are not replicable by discipline in a single chat window. The first is the **fresh-context critic**: the critic is spawned as a subagent in a new session that has not seen the back-and-forth producing the outline. The assumption behind this design — that a same-session critic softens its feedback on work it helped draft — is a bet grounded in the broader LLM sycophancy literature, not a benchmarked property of this tool specifically. It is an engineering bet the piece should name as such. The second is **artifact persistence**: outlines, research notes, and drafts on disk, versionable and referenceable without replaying context. The research artifact deserves a call-out of its own — `/critique` produces a `research.md` file recording the sources consulted, the prior art located, the counterarguments surfaced, and the facts verified, and that file has standalone value even if the outline it was gathered for gets abandoned. For engineers who have tried and failed to turn LLM chat into durable research notes, this is a feature on its own terms.

The speed objection is shape-dependent, and the piece will own that rather than reject it. For fluent writers, Prospero does not accelerate throughput. For writers whose baseline is closer to zero — the perfectionist who spends a month thinking about a piece and never starts; the engineer with three unfinished drafts in `drafts/` — it can move output from nothing to something, which is a substantial throughput improvement even if no individual piece is produced faster. The piece commits to this audience. The durable claim for both user profiles is rigor: the argument has been challenged before you publish it.

On the secondary flags: Perspectives and DebateDevil are monetized multi-persona debate apps that share a conceptual neighborhood but sit in a different category — Prospero is a Claude Code plugin focused on oppositional critique of the author's own argument, with fresh-context subagents and persistent artifacts, free and composable with the rest of the Claude Code workflow. Name them in a sentence and move on. On sycophancy: cite once, acknowledge it is well-traveled ground, and frame Prospero's contribution as applying the known finding structurally rather than re-arguing it. On Superpowers: say "demonstrated to practitioners' satisfaction" or similar; do not claim empirical proof. These are two-minute fixes that close easy HN hits.

## Entry Point
Brief personal frame that gets out of its own way. Roughly: I like writing but find it stressful, I hem and haw for weeks, and pieces often stall at 80%. I tend to think of myself as a better editor than an author. Prospero is the tool I built to synthesize those strengths — and to finish pieces I would not otherwise have finished. One short paragraph, maybe two, then pivot directly to the `/interrogate` → `/critique` → `/author` → `/revise` structure and the fresh-context-critic principle. The reader should know what Prospero *is* by the end of the second screen.

## Sections

### The author seat is stressful; the editor seat isn't
- **Claim:** A specific kind of writer — the engineer-essayist, the reluctant author who likes editing more than drafting, the perfectionist whose baseline output is closer to zero than they'd like to admit — has been badly served by the dominant shape of AI writing tools. The prose-machine model asks you to play author with a faster keyboard. Prospero inverts it: AI plays author, I play editor.
- **Evidence:** Personal frame, brief. Audience calibration: this profile is common on HN, where many technically strong writers post on the side, get paralyzed by the author seat, and have drafts that never ship. Cross-reference: echoes "AI like a bicycle" — the cyclist pedals, the bicycle amplifies; the author edits, the tool amplifies.
- **Objection:** This sounds like a narrow personal tool, not a library worth announcing.
- **Resolution:** It is a narrow tool, and that is the pitch. The next section makes the narrow audience's value concrete.

### What Prospero does, phase by phase
- **Claim:** Prospero stages writing as four discrete phases — interrogate, critique, author, revise — each solving a distinct problem and leaving durable artifacts behind. Each phase is a slash command in Claude Code; each phase produces or consumes a file on disk; critique runs in a fresh context window.
- **Evidence — phase distinctions:**
  - `/interrogate` — Socratic questioning to extract an argumentative outline from a topic. Produces `<drafts>/<slug>/outline.md` with thesis, steelmanned antithesis, synthesis, entry point, and section-level claims/evidence/objections/resolutions. Solves: the problem of starting with a fuzzy idea and hand-wringing indefinitely.
  - `/critique` — oppositional review of the outline (or later, the draft). Runs in a fresh context subagent that sees only the artifact. Produces a structured rubric-based critique (ratings, specific findings, summary judgment of "ready / needs revision / needs rethinking") and appends research findings to `research.md`. Solves: the author being too close to their own argument to see its weakest joints.
  - `/author` — writes the first-pass draft from the outline, using the author's configured voice guide and the research gathered by the critic. Produces the draft file at the configured post path. Solves: the friction of starting a blank page once the argument is settled.
  - `/revise` — collaborative line-level and structural review of a draft the author has already edited. Optionally re-invokes `/critique` in draft mode. Solves: the final pass where a human's tacit sense of voice and a machine's structural eye combine.
- **Evidence — interaction snippets:** Short walkthrough of one plausible topic through each phase, kept tight. Show what an interrogation question looks like, what a critique finding looks like, what the generated first-pass prose looks like before the author edits it, and how the research file accumulates. Treat snippets like API docs: enough shape to convey the thing, not enough to drown the argument.
- **Configuration:** Three files in `.prospero/` — `config.toml` (active preset), `voice.md` (author's voice rules), `audience.md` (reader description and research sources). Hugo preset bundled; presets for other static site generators extend the same scaffold.
- **Objection:** This sounds like a workflow, not a tool.
- **Resolution:** Hand off to the next section on why the workflow needs to be a tool.

### Why the dialectical shape needs tooling
- **Claim:** "Dialectical" here means nothing more complicated than "staged argument against itself" — an outline that names its own best counterargument, a critic that reviews the outline as an opponent rather than a collaborator. Two structural properties distinguish Prospero from a prompt template a disciplined user could operate in any chat window.
- **Evidence — fresh-context critic:** The critic is spawned in a new session with no exposure to the outline's drafting process. The design assumption is that a same-session model softens critique on its own prior output; this is a reasonable bet given the documented LLM sycophancy literature, not a property Prospero has benchmarked in isolation. Name this as an engineering commitment grounded in prior evidence, not as proven fact.
- **Evidence — persistent artifacts:** Outline, research, and draft live on disk as separate files. Each phase reads and writes what it needs without replaying the conversation that produced earlier artifacts. This is load-bearing for composition — the critic in a fresh context can still read the research the author produced manually; the author phase can incorporate critique findings without a 10,000-token conversational preamble.
- **Evidence — the Superpowers lineage:** Superpowers is the Claude Code plugin that popularized the staged-subagent-with-persistent-artifacts pattern; it has been widely and positively received by practitioners in the Claude Code ecosystem, though the evidence is experiential (practitioner reviews) rather than controlled comparison. Prospero applies that pattern to blog-post writing. One paragraph explaining what Superpowers does, what it has meant for the expectations of the space, and why Prospero inherits its design rather than reinventing it. Be careful with claims about what Superpowers "proved" — frame as "demonstrated to practitioners' satisfaction" or "the pattern has become the default expectation," not as empirical proof.
- **Evidence — guardrails:** The pattern guards against specific failure modes on both sides. Author failure modes: wanting to believe your thesis is strong ("sniffing your own farts"); rationalizing past weak joints; bringing flattened strawmen to the antithesis because you're not trying hard enough. LLM failure modes: sycophancy in-session; over-agreement in multi-turn chat; failure to steelman without explicit prompting. Each phase of Prospero is pointed at one of these.
- **Objection:** LLMs aren't actually good enough at oppositional critique for this to be reliable.
- **Resolution:** They are, when prompted accordingly and given a clean slate. This is a capability claim the piece rests on; it leans on the reader's own experience with tools like Superpowers rather than trying to re-prove the capability from scratch.

### Where it fits in an AI-amplified workflow
- **Claim:** The design principle behind Prospero is the one behind "AI like a bicycle" and "software architecture after AI": amplify the author's intent at the layer where effort still matters, and stop automating the layer AI has already made cheap. In writing, prose is the cheap layer; the argument is the expensive one. Most AI writing tools have it backwards.
- **Evidence:** Cross-reference "AI like a bicycle" — the bicycle requires skill, amplifies intent, does not substitute for pedaling. Cross-reference "Software architecture after AI" — the boundary of "hard to change" has moved in software; a similar boundary has moved in writing, and what remains hard is the strategic/argumentative layer. Pointing AI at the already-cheap layer wastes the amplification.
- **Parallel tools worth a sentence each:** Perspectives and DebateDevil are monetized apps with adjacent framing (AI as interlocutor, not writer). Prospero sits in a different category: a free Claude Code plugin focused on oppositional critique of the author's own argument, composable with the rest of a Claude Code workflow, with persistent file-based artifacts. Quick acknowledgment, specific differentiator, move on.
- **Objection:** The bicycle frame is graceful for personal use, but libraries get adopted because they save time, not because they embody a philosophy.
- **Resolution:** Agreed, and the piece does not oversell a productivity claim. Install pitch is pragmatic: if the editor seat is more comfortable for you than the author seat, or if perfectionism keeps your drafts unfinished, Prospero is built for that shape of writer.

### What it doesn't do
- **Claim:** Prospero does not make fluent writers faster. It does not finish a piece whose thesis is confused. It does not solve distribution. And it has a genuine design tension — blunting — that the piece names honestly and opens to scrutiny.
- **Evidence — impact honest-frame:** "Software architecture after AI" still took weeks to chew on; Prospero did not accelerate that piece, it gave me tools for strengthening it. I believe I would not have finished it without them. "Accumulated ignorance at scale" covers conceptually similar ground and was written with a weaker version of this process; the comparison makes the sharpening visible. (Specifics deferred to drafting; the comparison itself is committed.)
- **Evidence — blunting, in concrete terms:** Deflation is a feature, not a bug; the critic *should* puncture arguments that don't work, and I accept that cost. Blunting is the harder problem: the same critic can soften a declarative stance that was worth making sharply. In practice I mitigate this by reading every critique finding section-by-section and overriding any that would flatten a position I hold strongly — roughly, I accept findings that identify weak evidence, and I push back on findings that ask me to hedge an opinion. That judgment call is imperfect and I have gotten it wrong in both directions. The tool's prompts do not solve this; improving the balance between rigor and preservation of voice is a design question I would genuinely welcome outside perspective on.
- **Objection:** If it doesn't make fluent writers faster, what is the measurable benefit?
- **Resolution:** For fluent writers, the benefit is confidence and a higher argumentative floor — no throughput claim. For paralyzed perfectionists, the benefit is throughput, because zero to something is a substantial delta and the critic front-loads the "what will commenters say" anxiety that kept previous drafts unpublished.

### Install and try it
- **Claim:** Installing Prospero and producing a first outline should take under ten minutes if you already run Claude Code.
- **Evidence:** Install command (Claude Code plugin marketplace; repo link). `.prospero/` scaffold contents: `config.toml` pointing at a preset, `voice.md` template describing the author's voice rules, `audience.md` template describing the reader and listing research sources. Hugo preset works out of the box for Hugo users; other presets listed briefly. Minimal first-session walkthrough: `cd` to your blog repo, `/interrogate` to produce an outline from a topic, `/critique` for adversarial review, `/author` for a draft. The walkthrough example threads through a committed-to topic (to be chosen at authoring, ideally one whose thesis the piece can afford to make visible in public). Repo link prominent.
- **Objection:** The install footprint assumes Claude Code. What if I don't use Claude Code?
- **Resolution:** Prospero is tied to Claude Code because the subagent/skill model is load-bearing for the fresh-context critic. A port to another harness is conceivable but not this announcement; the ideas are portable, the implementation isn't.

## So What?
If the editor seat is easier for you than the author seat, or if your `drafts/` folder keeps growing because perfectionism stops you from shipping, Prospero is built for that shape of writer and "Software architecture after AI" is what it looks like when the shape works. Install it, run `/interrogate` on a topic you've been sitting on, see whether the critic's first pass tells you something you didn't know. The broader claim the tool embodies — amplify argument, not prose — is worth holding even if you never install anything, but the announcement's real ask is specific: try it on one piece.
